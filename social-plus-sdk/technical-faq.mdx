## Setup and Authentication

<AccordionGroup>
  <Accordion title="Which SDK should I start with?">
    Use the SDK when you need custom UI and app-owned product flows. Choose the platform SDK that matches your app runtime: iOS, Android, TypeScript, or Flutter.

    See [SDK Overview](/social-plus-sdk/overview) and [Installation & Quick Start](/social-plus-sdk/getting-started/overview).
  </Accordion>

  <Accordion title="Where do API key and region fit?">
    Initialize the SDK with the API key for your social.plus application and the region where that application was created. Keep production key management aligned with your app security model and avoid committing secrets.

    See [Installation & Quick Start](/social-plus-sdk/getting-started/overview).
  </Accordion>

  <Accordion title="Do SDK apps manage access tokens directly?">
    The SDK manages its session and access-token lifecycle after login. If secure mode is enabled, your backend provides an auth token and your app supplies a session handler or renewal callback where the platform SDK requires it.

    See [Authentication](/social-plus-sdk/getting-started/authentication) and [User Token Management](/social-plus-sdk/core-concepts/user-management/user-operations/user-token-management).
  </Accordion>

  <Accordion title="Can users browse without logging in?">
    Visitor mode is available for read-oriented browsing scenarios where supported. Use it only for screens that do not need authenticated actions such as posting, reacting, joining, or messaging.

    See [Visitor Mode](/social-plus-sdk/getting-started/visitor-mode).
  </Accordion>
</AccordionGroup>

## Data Loading and Realtime

<AccordionGroup>
  <Accordion title="What is the difference between a live object and a live collection?">
    A live object tracks one entity, such as a post, user, channel, message, community, or room. A live collection tracks a list, such as posts, comments, messages, members, followers, reactions, rooms, or notifications.

    See [Live Objects & Collections](/social-plus-sdk/core-concepts/realtime-communication/live-objects-collections/overview).
  </Accordion>

  <Accordion title="Why do I only see the first page of results?">
    Most list reads return a paged live collection or platform paging type. Render the current data, check whether a next page exists, and call the platform-specific next-page API when the user scrolls or requests more.

    See the live collection guide for your platform and the relevant query page, such as [Query Posts](/social-plus-sdk/social/content-management/posts/retrieval/query-posts), [Query Comments](/social-plus-sdk/social/content-management/comments/retrieval/query-comments), or [Query Messages](/social-plus-sdk/chat/messaging-features/messages/query-and-filter-messages).
  </Accordion>

  <Accordion title="Do live objects automatically mean cross-device realtime?">
    Live objects and collections observe SDK-managed state. Some SDKs or features also require explicit realtime topic subscription for cross-device updates. Follow the platform-specific realtime guide instead of assuming one subscription model across every SDK.

    See [Realtime Events](/social-plus-sdk/core-concepts/realtime-communication/realtime-events/overview).
  </Accordion>

  <Accordion title="What should I clean up when a screen is closed?">
    Clean up the platform-native subscription handle: unsubscribe callbacks in TypeScript, invalidate notification tokens in iOS, dispose Rx subscriptions or cancel coroutine collectors in Android, and cancel streams or dispose live collections in Flutter.

    See [Live Objects & Collections](/social-plus-sdk/core-concepts/realtime-communication/live-objects-collections/overview).
  </Accordion>
</AccordionGroup>

## Social Content

<AccordionGroup>
  <Accordion title="What do targetType and targetId mean for posts?">
    `targetType` selects the feed scope, usually a user feed or community feed. `targetId` identifies the user or community that owns the feed. Use the post creation page for the post type you are building.

    See [Posts Overview](/social-plus-sdk/social/content-management/posts/overview).
  </Accordion>

  <Accordion title="How should I choose a post creation page?">
    Match the SDK post type to the content you are creating: text, image, video, file, audio, clip, poll, room, live stream, custom, or mixed media. Platform support is not identical, so read the platform notes on the specific page.

    See [Content Management](/social-plus-sdk/social/content-management/overview).
  </Accordion>

  <Accordion title="How do media uploads fit into post and message creation?">
    Upload or obtain the media/file object first where the SDK requires it, then pass the resulting file, file ID, URI, or platform-specific media data into the post or message creation API.

    See [File Handling](/social-plus-sdk/core-concepts/content-handling/files-images-and-videos/file), [Image Handling](/social-plus-sdk/core-concepts/content-handling/files-images-and-videos/image-handling), and [Video Handling](/social-plus-sdk/core-concepts/content-handling/files-images-and-videos/video-handling).
  </Accordion>

  <Accordion title="Can I assume comments, reactions, mentions, and moderation work the same on every platform?">
    No. The shared product concepts are similar, but each SDK exposes native method names, builders, return types, and support gaps. Use the platform-specific snippets and notes on each page.

    See [Comments Overview](/social-plus-sdk/social/content-management/comments/overview), [Reactions](/social-plus-sdk/core-concepts/content-handling/reactions), [Mentions](/social-plus-sdk/core-concepts/content-handling/mentions), and [Content Flagging](/social-plus-sdk/social/content-management/moderation/content-flagging).
  </Accordion>
</AccordionGroup>

## Chat

<AccordionGroup>
  <Accordion title="What is the main chat data model?">
    Chat is organized around channels, members, messages, and unread or receipt state. Channel and message APIs are platform-specific, so start with the chat overview and then use the exact page for the operation you need.

    See [Chat Overview](/social-plus-sdk/chat/overview).
  </Accordion>

  <Accordion title="How do I create and send messages?">
    Create or retrieve the target channel, then use the message creation page for the message type you need. Text, image, file, video, audio, custom, and reply flows have separate platform notes.

    See [Messaging Features](/social-plus-sdk/chat/messaging-features/overview).
  </Accordion>

  <Accordion title="How should unread counts and read receipts be handled?">
    Use the SDK unread and receipt APIs instead of deriving state only from local UI. Read, delivered, and receipt-sync surfaces differ by platform and by channel/subchannel concept.

    See [Unread Status](/social-plus-sdk/chat/engagement-features/unread-status/overview).
  </Accordion>

  <Accordion title="Where do channel membership and governance actions live?">
    Channel join/leave, member queries, preview members, roles, bans, and mutes live under conversation management. Treat these as SDK operations, while app policy decides who is allowed to see or trigger each action.

    See [Conversation Management](/social-plus-sdk/chat/conversation-management/overview).
  </Accordion>
</AccordionGroup>

## Platform Differences

<AccordionGroup>
  <Accordion title="Can I copy a snippet from one platform to another?">
    Use snippets as platform-specific examples, not as cross-platform contracts. The same capability may use callbacks in TypeScript, async or live objects in iOS, Rx or Paging in Android, and Dart streams or builders in Flutter.
  </Accordion>

  <Accordion title="How do I know whether a feature exists on my SDK?">
    Check the platform notes on the feature page and the platform changelog. If a page says a feature is not exposed on a platform, do not build against an inferred method name.

    See [SDK Changelogs](/social-plus-sdk/changelogs/android-sdk-changelog).
  </Accordion>

  <Accordion title="How should AI assistants use these docs?">
    Ask the assistant to cite the exact SDK page and platform before generating code. It should not infer missing SDK APIs from another platform or from REST API documentation.
  </Accordion>
</AccordionGroup>

## Troubleshooting

| Symptom | First checks |
| --- | --- |
| Login fails | Confirm API key, region, user ID, secure-mode token flow, and session handler behavior. |
| A list never loads more | Check the live collection or paging API for next-page availability and lifecycle cleanup. |
| UI does not update after another device changes data | Confirm realtime subscription requirements for the platform and feature. |
| A post or message type is missing | Verify platform support on the exact creation page and changelog before assuming parity. |
| Media creation fails | Confirm upload step, returned file or media type, size/format requirements, and platform-specific attachment builder. |
| Unread counts look stale | Use SDK read/delivered/receipt-sync APIs and verify channel or subchannel scope. |

## Escalation Checklist

When the docs do not answer your issue, open a forum or support thread with:

- SDK platform and version
- App platform version and runtime
- Region
- Feature page you followed
- Minimal code shape or operation sequence
- Expected result and actual result
- Error codes, timestamps, and safe entity IDs where available
- Whether the same issue reproduces after a clean login/session renewal
