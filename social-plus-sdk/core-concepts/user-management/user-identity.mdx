Social+ SDK uses a **decentralized user management approach** that gives you complete control over your user data while providing powerful social features.

## Core Principles

<Info>
**No User Data Storage**: Social+ SDK does not store or manage any personal user data. This means no user profiles, emails, or personal information is stored in Social+ systems.
</Info>

### Key Benefits

<CardGroup cols={3}>
<Card title="Privacy First" icon="shield-check">
Your users' data stays in your systems, ensuring privacy and compliance
</Card>

<Card title="No Migration" icon="database">
Integrate Social+ without importing or migrating existing user profiles
</Card>

<Card title="Full Control" icon="gear">
Handle user management with your existing application logic and systems
</Card>
</CardGroup>

## User Identification

Every user in Social+ is represented by a unique `userId` - a string that uniquely identifies the user and remains immutable throughout their lifetime.

### Choosing the Right User ID

<Tabs>
<Tab title="✅ Recommended">
**Database Primary Keys**
```
userId: "12345"
userId: "user_abc123"
userId: "uuid-4f8b-4c2d-9e1a-8b7c6d5e4f3g"
```

**Benefits:**
- Immutable throughout user's lifetime
- Unique by design
- Already exists in your system
- Consistent across your application

**Examples:**
- Auto-incrementing integers
- UUIDs
- Custom-generated unique identifiers
</Tab>

<Tab title="❌ Not Recommended">
**Changeable Identifiers**
```
userId: "john.doe@email.com"    // Email can change
userId: "johndoe123"            // Username can change
userId: "john_doe"              // Display name can change
```

**Why to Avoid:**
- Can change over time
- User may want to update them
- May not be unique across time
- Can cause data consistency issues
</Tab>
</Tabs>

<Warning>
**Important**: Once a user is created in Social+, their `userId` cannot be changed. Choose your user identification strategy carefully.
</Warning>

## User Roles & Permissions

Enhance user capabilities by assigning roles with specific permissions defined in your admin panel.

<Info>
**Role-Based Access Control**: Social+ uses a comprehensive permission system that allows you to create hierarchical moderation structures and fine-grained access control.
</Info>

### Quick Overview

<CardGroup cols={2}>
<Card title="Default Roles" icon="users">
**Built-in Role Hierarchy**  
Member → Channel Moderator → Community Moderator → Super Moderator → Global Admin
</Card>

<Card title="Custom Roles" icon="settings">
**Flexible Permissions**  
Create custom roles with specific permissions in the Social+ Console
</Card>
</CardGroup>

### Basic Role Assignment

```javascript
// Assign roles during login
await client.login({
  userId: 'user123',
  displayName: 'John Doe',
  roles: ['moderator', 'vip'] // Define roles in admin panel
});
```

<Note>
**Learn More**: For comprehensive role management, permission checking, and implementation examples, see the dedicated [Roles & Permissions](/social-plus-sdk/core-concepts/user-management/roles-permissions) guide.
</Note>

## User Repository

While Social+ doesn't store user profiles, it provides tools for querying and searching users within your social features.

### Initializing User Repository

<CodeGroup>
```swift iOS
let userRepository = AmityUserRepository(client: client)
```

```kotlin Android
fun initUserRepository() {
    val userRepository = AmityCoreClient.newUserRepository()
}
```

```typescript TypeScript
import { UserRepository } from '@amityco/ts-sdk';
```

```dart Flutter
void initUserRepository() {
  final userRepository = AmityCoreClient.newUserRepository();
}
```
</CodeGroup>

## Best Practices

<CardGroup cols={2}>
<Card title="User ID Strategy" icon="key">
**Do:**
- Use database primary keys
- Ensure immutability
- Keep IDs consistent across your app

**Don't:**
- Use changeable identifiers
- Include personal information
- Create complex ID schemes
</Card>

<Card title="Data Management" icon="database">
**Do:**
- Keep user profiles in your system
- Sync display names as needed
- Use metadata for Social+ specific data

**Don't:**
- Store personal data in Social+
- Rely on Social+ for user management
- Ignore privacy compliance
</Card>

<Card title="Role Management" icon="user-shield">
**Do:**
- Define clear role hierarchies
- Use roles for feature access
- Plan for role changes

**Don't:**
- Hardcode role permissions
- Create too many role levels
- Forget to update roles
</Card>

<Card title="Performance" icon="zap">
**Do:**
- Cache user repository queries
- Implement pagination
- Use efficient search patterns

**Don't:**
- Make unnecessary API calls
- Load all users at once
- Ignore query optimization
</Card>
</CardGroup>
