**Why the large type count drop is a feature, not a bug:**

The regex extractor walks all `.kt` and `.java` files in the SDK repo and captures any `public` class/interface it finds via regex — regardless of whether that type is actually in the SDK's public API or is internal/vendor code. The result is severe over-counting.

Dokka GFM applies the same package-level suppression as the `dokkaHtml` config in `amity-sdk/build.gradle` (applied manually in the extractor since `perPackageOption` only applies to `dokkaHtml`, not `dokkaGfm`):
- `com.amity.socialcloud.sdk.chat.*` — suppressed (internal data/domain layer)
- `com.amity.socialcloud.sdk.common.*` — suppressed
- `com.amity.socialcloud.sdk.core.*` — suppressed
- `com.amity.socialcloud.sdk.entity.*` — suppressed
- `com.amity.socialcloud.sdk.infra.*` — suppressed
- `com.amity.socialcloud.sdk.social.*` — suppressed
- `com.amity.socialcloud.sdk.video.*` — suppressed
- `com.ekoapp.*` — suppressed (legacy namespace)
- `org.amity.*` — suppressed

The Dokka extractor includes only: `com.amity.socialcloud.sdk.api.*`, `com.amity.socialcloud.sdk.model.*`, `com.amity.socialcloud.sdk.helper.*`.

## 4. Same name, different member shape

For types both extractors captured, Dokka consistently provides **more members** (via builder property surfacing and implicit-public Kotlin conventions) rather than fewer.

No cases where regex has a member Dokka is missing on a shared type in the `api.*`/`model.*` namespace — i.e., there are no cases where Dokka under-reports on confirmed-public types. The 3,885 regex-only members are all attached to regex-only types (false positive types).

## 5. Coverage analysis (doc-as-test analog)

Android doc-as-test currently runs clean (no failing compile checks), unlike iOS which had 42 signature findings at the time of the DocC migration. The comparison question becomes: would Dokka catch issues the regex extractor is structurally blind to?

**Yes, for two categories:**

**A. Builder API properties** — The 177 Dokka-only members on shared types (e.g., `AmityCommentCreator.commentId`) are surfaced by Dokka but not regex. If a doc page references `AmityCommentCreator.commentId` (a valid builder property), the regex-based validator would flag it as unknown. With Dokka as primary, those references correctly pass validation.

**B. Kotlin implicit-public conventions** — Dokka understands that Kotlin class members without access modifiers are `public` by default. Regex required an explicit `public` keyword. This means the regex extractor has been producing false-positive drift flags for valid Kotlin-convention API (like `apiKey` companion properties on enum classes).

**Flow/LiveData analog (iOS Combine analog)**: The iOS migration found 42 `$publisher` (Combine) references that regex missed. The Android equivalent would be `Flow<T>` and `LiveData<T>` return types. Dokka captures full signatures (parameter/return types); regex only captures member names. Once the swap is done and the signature field is activated in validation, Flow/LiveData type drift would be detectable — this is the equivalent of the iOS Combine win.

## 6. Recommendation

**Proceed to swap `android.json` to Dokka-sourced output (task 0079).**

Rationale:
1. **Zero false positives** on the core `api.*`/`model.*` surface — the 726 regex-only types are all vendor code or internal packages
2. **177 net new members** on confirmed-public types that regex was missing — these eliminate false-positive drift warnings in the validator
3. **2 new types** (`AmityRoomSortOption`, `AmityRoomStatus`) recovered
4. **Aligned with SDK team intent** — Dokka's suppression rules directly encode what the SDK team considers public API; we're inheriting their judgment rather than guessing from file paths

The only pre-swap action: verify that the 177 Dokka-only members on shared types (builder properties, `apiKey` companions) are intentionally public and not accidentally surfaced. A spot-check of `AmityCommentCreator` and `AmityChannelFilter` source confirms they are — Kotlin implicit-public convention.

**Caveat**: The `android-accuracy-validator.py` will need a path-exclusion and platform-type allowlist update (similar to iOS task 0077) after the swap, since the validator was calibrated against the regex surface. File a follow-up task for that.
