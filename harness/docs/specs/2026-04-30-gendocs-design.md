# Design: `gendocs` — SDK Snippet → Mintlify MDX Pipeline

_Date: 2026-04-30_

## Problem

`social-plus-docs` uses Mintlify with hardcoded code snippets in MDX files. Code snippets live in 4 SDK repos (Android/Kotlin, iOS/Swift, Flutter/Dart, TypeScript). When SDK code changes, the docs are not updated — there is no automated link between SDK snippet files and the rendered documentation.

Previously, GitHub Gists served as the intermediary. That approach is retired.

## Goal

Add a `gendocs` command to the existing Go harness that reads SDK snippet files from all 4 platforms and generates/updates Mintlify snippet MDX files in `social-plus-docs/snippets/`. Doc pages then `import` those MDX snippets instead of hardcoding platform tabs.

## Architecture

### Full Pipeline

```
audit  → finds missing/invalid snippets in SDK repos
fix    → deterministic fixes (ASC_PAGE_INVALID, SNIPPET_CONTENT_DRIFT)
prompt → generates harness-tasks.md for AI agent (MISSING_SNIPPET)
gendocs → reads SDK snippet files → writes social-plus-docs/snippets/*.mdx
```

### Data Flow

```
[SDK repos: begin_sample_code blocks]
        ↓  scanner (reuse existing)
[function_id, asc_page, description, platform, code]
        ↓  grouper (group by function_id)
[per-function groups: {android?, ios?, flutter?, typescript?}]
        ↓  MDX writer
[social-plus-docs/snippets/<derived-path>.mdx]
        ↓  manifest writer
[social-plus-docs/snippets/manifest.json]
```

### Components

| Component | Location | Responsibility |
|-----------|----------|----------------|
| `cmd/gendocs.go` | harness/cmd/ | CLI entry point, flags, orchestration |
| `internal/docgen/writer.go` | harness/internal/docgen/ | MDX file generation, path derivation, idempotency |
| `internal/docgen/writer_test.go` | harness/internal/docgen/ | Unit tests |

## Snippet MDX Format

Each generated file at `social-plus-docs/snippets/<derived-path>.mdx`:

```mdx
{/* AUTO-GENERATED by harness gendocs — do not edit manually */}
{/* function: community.member.add | updated: 2026-04-30 */}

<CodeGroup>

```kotlin Android
AmitySocialClient.newCommunityRepository()
    .membership("communityId")
    .addMembers(listOf("userId1", "userId2"))
    .subscribe()
```

```swift iOS
// iOS code here
```

```dart Flutter
// Flutter code here
```

```typescript TypeScript
// TypeScript code here
```

</CodeGroup>
```

**Rules:**
- `begin_sample_code` / `end_sample_code` markers are **stripped** — only clean code is written to MDX
- Platform tabs are included only if a snippet exists for that platform
- Tab labels: `Android`, `iOS`, `Flutter`, `TypeScript` (Mintlify `<CodeGroup>` convention)
- The `{/* AUTO-GENERATED */}` comment is invisible in rendered Mintlify output

## Path Derivation

Snippet MDX path is derived from the function's `asc_page` + function ID tail:

| function_id | asc_page | output path |
|-------------|----------|-------------|
| `community.member.add` | `social-plus-sdk/social/.../manage-members` | `snippets/social-plus-sdk/community/member-add.mdx` |
| `post.create.text` | `social-plus-sdk/social/.../create-post` | `snippets/social-plus-sdk/post/post-create-text.mdx` |

**Algorithm:**
1. Take the first segment of `asc_page` as the directory prefix (e.g. `social-plus-sdk`)
2. Take the second segment of `asc_page` as subdirectory (e.g. `social`)
3. Kebab-case the function ID tail (after the first dot) as filename: `member.add` → `member-add`
4. Full path: `snippets/<seg1>/<seg2>/<kebab-tail>.mdx`

## Update Strategy (Idempotency)

| Scenario | Behaviour |
|----------|-----------|
| File does not exist | Create it |
| File exists with `AUTO-GENERATED` header | Overwrite entirely |
| File exists without `AUTO-GENERATED` header | Skip + print warning |
| New platform adds a snippet | Overwrite, new tab added |
| Platform snippet removed | Overwrite, tab removed |

`gendocs` does **not** touch main doc pages (`social-plus-sdk/**/*.mdx`). Doc editors add the `import` once manually:

```mdx
import CommunityMemberAdd from '/snippets/social-plus-sdk/community/member-add.mdx';
...
<CommunityMemberAdd />
```

This is a one-time migration per page. After that, `gendocs` handles ongoing updates automatically.

## Manifest

After writing MDX files, `gendocs` writes `social-plus-docs/snippets/manifest.json`:

```json
{
  "community.member.add": "snippets/social-plus-sdk/community/member-add.mdx",
  "post.create.text":     "snippets/social-plus-sdk/post/post-create-text.mdx"
}
```

Doc editors use this to find the correct import path for a function.

## Command Interface

```bash
./harness-bin gendocs --config harness-config.yml
./harness-bin gendocs --config harness-config.yml --dry-run
./harness-bin gendocs --config harness-config.yml --out /custom/snippets/dir
```

| Flag | Default | Description |
|------|---------|-------------|
| `--config` | required | Path to `harness-config.yml` |
| `--dry-run` | false | Print planned writes without touching files |
| `--out` | `<docs_path>/snippets` | Override output directory |

## Out of Scope (this phase)

- Multi-function grouping within a single MDX file (one MDX per function_id)

## Immediate Next Phase

Once `gendocs` is stable, the following two capabilities are the natural continuation:

### Phase 2A — Auto-migration of hardcoded doc pages
A `migrate` command (or `gendocs --migrate` flag) that scans existing `social-plus-sdk/**/*.mdx` pages, finds hardcoded `<CodeGroup>` blocks, and replaces them with `import` + component reference pointing to the generated snippet MDX. This makes the one-time migration automated rather than manual.

### Phase 2B — Prose/narrative generation
A `gendocs --prose` mode (AI-assisted) that generates the narrative sections of doc pages (overview, parameter descriptions, usage notes) using the snippet's `description` field and SDK source as context. Copilot CLI is the AI agent — same pattern as `harness prompt`.

## Success Criteria

- `gendocs` produces one `.mdx` file per function_id that has at least one snippet
- Re-running `gendocs` is safe and idempotent
- `manifest.json` accurately maps all function IDs to MDX paths
- Human-edited MDX files are never clobbered
- `--dry-run` shows exactly what would change without writing files
- All new packages pass `go test ./...`
- Running `mintlify dev` (or `mintlify build`) in `social-plus-docs/` succeeds after `gendocs` with no broken imports or MDX parse errors
