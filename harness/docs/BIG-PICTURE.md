## The Problem

The Social Plus SDK documentation lives in Mintlify (`social-plus-docs`).
The actual SDK code lives in **4 separate repos** across 4 platforms:

| Platform | Repo | Language |
|----------|------|----------|
| Android | `Amity-Social-Cloud-SDK-Android` | Kotlin |
| iOS | `AmitySDKIOS` | Swift |
| Flutter | `Amity-Social-Cloud-SDK-Flutter-Internal` | Dart |
| TypeScript | `AmityTypescriptSDK` | TypeScript |

Today, every code example in the docs is **hardcoded** ‚Äî copy-pasted into MDX files as raw fenced code blocks inside `<CodeGroup>` components. This means:

- When SDK code changes, docs silently go stale (nobody notices)
- Adding a code example to docs requires a separate manual copy-paste PR
- There's no guarantee the hardcoded code is correct ‚Äî it can't be compiled or tested
- Different sections of the same page can have inconsistent platform coverage

## The Harness CLI (`harness-bin`)

The harness is a Go CLI with 5 commands:

### `audit` ‚Äî Find problems
Scans all 4 SDK repos + all doc pages. Produces `harness-report.json` with findings:

| Finding Type | Meaning |
|---|---|
| `MISSING_SNIPPET` | Public SDK function has no snippet file in the SDK repo |
| `ASC_PAGE_INVALID` | Snippet's `sp_docs_page` is a broken/unknown URL |
| `DOC_MISSING` | Doc page referenced by snippet doesn't exist |
| `DOC_PAGE_STALE_IMPORT` | Doc page has a generated snippet but still uses hardcoded `<CodeGroup>` |

### `fix` ‚Äî Auto-repair what can be fixed
Normalizes `sp_docs_page` URLs (e.g. `https://docs.amity.co/...` ‚Üí relative path).
Flags findings that need human review.

### `gendocs` ‚Äî Generate snippet MDX files
Reads snippet markers from all 4 SDK repos ‚Üí writes `snippets/*.mdx` files.
One MDX file per function, containing all platform tabs.

```bash
./harness-bin gendocs --config harness-config.yml --clean
# --clean removes stale AUTO-GENERATED files before regeneration
```

### `migrate` ‚Äî Update doc pages to use generated snippets
Reads `harness-report.json` ‚Üí for each `DOC_PAGE_STALE_IMPORT` finding:
- Adds `import LoginUser from '/snippets/.../login-user.mdx';` at top of MDX
- Replaces `<CodeGroup>` block with `<LoginUser />` component

### `prompt` ‚Äî Generate AI task list
Reads unresolved `MISSING_SNIPPET` findings ‚Üí writes `harness-tasks.md` with
one AI-ready task per missing snippet (platform + function signature + context).

## Snippet Marker Format (in SDK repos)

Every code snippet is wrapped with markers:

```kotlin
/* begin_sample_code
   filename: AmityClientLogin.kt
   sp_docs_page: social-plus-sdk/getting-started/authentication
   sp_docs_section: step-2-login-user        ‚Üê NEW (Option A ‚Äî in progress)
   description: Login a user with auth token
   */
fun login(userId: String, authToken: String) {
    AmityCoreClient.login(userId)
        .authToken(authToken)
        .submit()
}
/* end_sample_code */
```

| Field | Required | Meaning |
|---|---|---|
| `filename` | ‚úÖ | Filename used as function key (drives output path) |
| `sp_docs_page` | ‚úÖ | Which doc page this snippet belongs to (docs.json relative path) |
| `sp_docs_section` | üîÑ planned | Which **section/CodeGroup** on that page (slug of `###` heading) |
| `description` | ‚úÖ | One-line description shown in generated MDX |

> **Legacy note:** `asc_page:` is still parsed as a backward-compat alias for `sp_docs_page:`.
> `gist_id:` has been removed.

## The Known Gap: Multiple CodeGroups Per Page

Some doc pages (like `authentication.mdx`) have **many `<CodeGroup>` blocks** ‚Äî one per section:

- Step 1: Initialize SDK
- Step 2: Login User
- Step 3: Check Auth Status
- Step 4: Logout
- Advanced: Session Handlers
- Alternative: Login with Access Token

The current `sp_docs_page` field only says "this snippet belongs to the authentication page" ‚Äî it doesn't say **which section/CodeGroup** it targets. This means:

- ~19 snippets all point to the same page
- The `migrate` command can't know which `<CodeGroup>` to replace with which snippet
- Blindly replacing the first `<CodeGroup>` would be wrong

### The Fix: `sp_docs_section` field (Option A ‚Äî next up)

Add a `sp_docs_section` field to snippet markers = the slugified form of the `###` heading above the target `<CodeGroup>`:

```kotlin
sp_docs_section: step-2-login-user  // matches "### Step 2: Login User"
```

The migrate command will:
1. Parse the MDX page's `###` headings
2. For each generated snippet, find the `<CodeGroup>` after the matching heading
3. Replace only that specific CodeGroup

This requires a backfill across ~1,438 snippets in all 4 SDK repos.

## What "Done" Looks Like

When this project is complete:

1. **Every SDK function** has a snippet file in the SDK repo with correct `sp_docs_page` + `sp_docs_section` metadata
2. **Every doc page CodeGroup** imports a generated snippet instead of hardcoding code
3. **CI runs the harness** on every SDK PR ‚Üí regenerates snippets ‚Üí opens a docs PR if anything changed
4. **SDK developers** own the code examples; docs stay fresh automatically
5. **Doc authors** write prose; they never copy-paste code again

## Full Pipeline (Current)

```bash
# 1. Find all problems
./harness-bin audit --config harness-config.yml

# 2. Auto-fix broken sp_docs_page URLs
./harness-bin fix --config harness-config.yml

# 3. Regenerate all snippet MDX files (clean slate)
./harness-bin gendocs --config harness-config.yml --clean

# 4. Re-audit to get fresh DOC_PAGE_STALE_IMPORT list
./harness-bin audit --config harness-config.yml

# 5. Update doc pages to use generated snippets (when sp_docs_section is ready: section-level)
./harness-bin migrate --config harness-config.yml

# 6. Generate AI task list for SDK devs to fill in missing snippets
./harness-bin prompt --config harness-config.yml
# ‚Üí produces harness-tasks.md
```
