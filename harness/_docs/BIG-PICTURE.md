# Social Plus SDK Docs вАФ Big Picture

> **Purpose:** A shared reference for what we're building, why, and where we are.
> Keep this updated as the project evolves.

---

## The Problem

The Social Plus SDK documentation lives in Mintlify (`social-plus-docs`).
The actual SDK code lives in **4 separate repos** across 4 platforms:

| Platform | Repo | Language |
|----------|------|----------|
| Android | `Amity-Social-Cloud-SDK-Android` | Kotlin |
| iOS | `AmitySDKIOS` | Swift |
| Flutter | `Amity-Social-Cloud-SDK-Flutter-Internal` | Dart |
| TypeScript | `AmityTypescriptSDK` | TypeScript |

Today, every code example in the docs is **hardcoded** вАФ copy-pasted into MDX files as raw fenced code blocks inside `<CodeGroup>` components. This means:

- When SDK code changes, docs silently go stale (nobody notices)
- Adding a code example to docs requires a separate manual copy-paste PR
- There's no guarantee the hardcoded code is correct вАФ it can't be compiled
- Different sections of the same page can have inconsistent platform coverage
- No machine-readable record of what code groups a page *should* have

---

## The Vision

**Code snippets live in the SDK repos. A page manifest defines what each doc page needs. The harness keeps everything in sync automatically.**

```
SDK repos (code source of truth)    Docs repo (spec + output)
вФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФА   вФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФАвФА
AmityClientLogin.kt                 authentication.manifest.yml
  /* begin_sample_code                sections:
     filename: AmityClientLogin.kt     login-user:
     sp_docs_page: .../auth              heading: "Step 2: Login User"
     description: Login a user           functions: [AmityClientLogin]

  fun login(userId: String) { ... }   snippets/getting-started/
  /* end_sample_code */                 login-user.mdx  вЖР AUTO-GENERATED

                                     authentication.mdx
                                       import LoginUser from '.../login-user.mdx'
                                       ### Step 2: Login User
                                       <LoginUser />    вЖР generated import
```

When an SDK developer updates the login function:
1. They update the snippet in the SDK repo
2. CI runs the harness вЖТ regenerates `login-user.mdx`
3. The doc page automatically shows the new code вАФ no human intervention

## Two Marker Types in SDK Repos

The SDK repos use **two distinct marker systems**:

### `begin_public_function` вАФ Public API coverage tracking
Marks a function as part of the public SDK API. Used to detect which functions are
*missing* a documentation snippet.

```kotlin
/* begin_public_function
   id: user.login
   api_style: async
   */
fun login(userId: String, authToken: String): Completable { ... }
/* end_public_function */
```

The `audit` command checks every function in public `*Repository`/`*Client` classes for this marker.
Functions missing the marker are reported as `PUBLIC_FUNC_UNANNOTATED`. The `annotate` command
auto-generates insertion patches; the `annotate --apply` command inserts them into SDK source.

### `begin_sample_code` вАФ Actual documentation snippets
Marks a self-contained code example that should appear in the docs.

```kotlin
/* begin_sample_code
   filename: AmityClientLogin.kt
   sp_docs_page: social-plus-sdk/getting-started/authentication
   description: Login a user with auth token
   */
fun login(userId: String, authToken: String) {
    AmityCoreClient.login(userId)
        .authToken(authToken)
        .submit()
}
/* end_sample_code */
```

| Field | Required | Meaning |
|---|---|---|
| `filename` | вЬЕ | Function key вАФ drives `snippets/` output path and import name |
| `sp_docs_page` | вЬЕ | Which doc page this snippet belongs to (docs.json relative path) |
| `description` | вЬЕ | One-line description in generated MDX |

**A `begin_public_function` without a matching `begin_sample_code` = `MISSING_SNIPPET` finding.**

> **Legacy note:** `asc_page:` is still accepted as a backward-compat alias for `sp_docs_page:`.

## Page Manifests вАФ Source of Truth Per Page

A **page manifest** (`*.manifest.yml`) lives alongside each doc page MDX file.
It defines the authoritative spec for what code groups a page needs:

```yaml
# social-plus-sdk/getting-started/authentication.manifest.yml
sections:
  initialize-sdk:
    heading: "Step 1: Initialize SDK"
    functions: [AmityClientSetup]
  login-user:
    heading: "Step 2: Login User"
    functions: [AmityClientLogin]
  check-auth-status:
    heading: "Step 3: Check Auth Status"
    functions: [AmityAuthStatus]
  logout:
    heading: "Step 4: Logout"
    functions: [AmityClientLogout]
```

**Why manifests (not `sp_docs_section` in SDK files):**
- Doc structure (headings, sections) belongs in the **docs repo**, not SDK repos
- A heading rename updates 1 file, not 1,400+ SDK snippet files
- Manifests are machine-readable вЖТ AI agents get precise, structured context
- Manifests drive both the harness audit and the AI task generation

## Computational vs Inferential Harness

This is the core design principle. Every step in the pipeline is classified as one or the other:

| Type | Who runs it | Characteristics |
|---|---|---|
| **Computational** | Machine / CI | Deterministic, rule-based, fast, repeatable вАФ pure Go code |
| **Inferential** | AI agent / human | Requires understanding, judgment, or code generation |

The harness CLI handles all computational steps. The AI agent (reading `harness-tasks.md`)
handles all inferential steps. They hand off to each other between phases.

---

## Three-Gate Validation Framework

The harness enforces quality through three sequential gates. Each gate must be green before
the next phase begins:

### Gate 1 вАФ Computational CI Gate вЪЩпЄП
Runs automatically; blocks merges on failure.

- `audit` shows **0 open findings** (all finding types, including `SNIPPET_KEY_PLATFORM_CONFLICT`)
- Finding: `SNIPPET_KEY_PLATFORM_CONFLICT` вАФ detected when sibling SDK platforms annotate the
  same snippet key with different `sp_docs_page` values. Fixed by aligning non-android platforms
  to Android's canonical path (Android wins by platform sort priority: android=0, ios=1, flutter=2, typescript=3).

### Gate 2 вАФ Inferential + Computational Oracle рЯ§Ц + вЪЩпЄП
Validates AI-generated work before it ships.

- `annotate` вЖТ generates patches with **`confidence: high | medium | low`** field:
  - `high` вАФ sibling platforms already document this function (safe to bulk-approve)
  - `low` вАФ no sibling confirmation; human review required
- `gendocs` вЖТ regenerate snippet MDX
- `audit` вЖТ verify 0 open findings remain

### Gate 3 вАФ Human Triage рЯ§Ц (future: automated LLM agent)
For issues that cannot be resolved computationally:
- SDK gaps (function doesn't exist on a platform) вЖТ mark `DOES_NOT_EXIST`
- New doc pages needed вЖТ create MDX + manifest
- Unresolvable `ASC_PAGE_INVALID` / `DOC_MISSING` conflicts вЖТ human judgment

Gate 3 is currently a manual step; designed to be automatable by an LLM agent in the future.

## The Harness CLI (`harness-bin`)

The harness is a Go CLI with 7 commands:

### `audit` вАФ Find problems вЪЩпЄП Computational
Scans SDK repos + doc pages + manifests. Produces `harness-report.json`:

| Finding Type | Meaning | Resolution |
|---|---|---|
| `MISSING_SNIPPET` | A manifest function has no snippet in any SDK repo | рЯ§Ц Inferential вАФ AI writes the snippet |
| `ASC_PAGE_INVALID` | Snippet's `sp_docs_page` doesn't match any page in docs.json | вЪЩпЄП Computational (auto-fix) or рЯ§Ц Inferential (needs_human) |
| `DOC_MISSING` | Doc page referenced by a snippet doesn't exist | рЯ§Ц Inferential вАФ human or AI creates the page |
| `DOC_PAGE_STALE_IMPORT` | Doc page still uses hardcoded `<CodeGroup>` instead of a generated import | вЪЩпЄП Computational вАФ `migrate` handles this |
| `PUBLIC_FUNC_UNANNOTATED` | Public function in `*Repository`/`*Client` has no `begin_public_function` | вЪЩпЄП Semi-computational вАФ `annotate --apply` inserts stubs; рЯ§Ц Inferential review for `id:` values |
| `SNIPPET_KEY_PLATFORM_CONFLICT` | Two or more platforms annotate the same snippet key with different `sp_docs_page` values | вЪЩпЄП Computational вАФ align non-android platform to Android's canonical path (Gate 1) |

### `fix` вАФ Auto-repair broken URLs вЪЩпЄП Computational
Normalizes `sp_docs_page` URLs (e.g. `https://docs.amity.co/...` вЖТ relative path).
Findings it cannot resolve algorithmically are marked `needs_human` вЖТ рЯ§Ц Inferential step.

### `gendocs` вАФ Generate snippet MDX files вЪЩпЄП Computational
Reads `begin_sample_code` markers from all 4 SDK repos вЖТ writes `snippets/*.mdx` files.
One MDX file per function, containing all available platform tabs.

```bash
./harness-bin gendocs --config harness-config.yml --clean
# --clean removes stale AUTO-GENERATED files before regeneration
```

### `migrate` вАФ Wire doc pages to generated snippets вЪЩпЄП Computational
Reads `harness-report.json` вЖТ for each `DOC_PAGE_STALE_IMPORT` finding:
- Adds `import LoginUser from '/snippets/.../login-user.mdx';` at the top of the MDX
- Replaces the matching `<CodeGroup>` block with `<LoginUser />`
- Uses the page manifest for section-level targeting (which CodeGroup to replace)

> When no manifest exists yet, section targeting requires judgment вЖТ рЯ§Ц Inferential:
> the AI reads the page and decides which heading a snippet belongs under.

### `prompt` вАФ Generate the AI agent runbook вЪЩпЄП Computational (output triggers рЯ§Ц Inferential)
Reads unresolved findings вЖТ writes `harness-tasks.md` with one AI-ready task per
missing snippet (platform, function, page context, manifest section).
The harness builds the runbook computationally; an AI agent executes it inferentially.

### `annotate` вАФ Generate annotation patches вЪЩпЄП Computational + рЯ§Ц Review
Reads `PUBLIC_FUNC_UNANNOTATED` findings вЖТ writes `annotation-patches.yml`.
Each patch contains: source file path, function name, suggested `id:`, insertion line,
and a **`confidence: high | medium | low`** signal (Gate 2):

- `confidence: high` вАФ 2+ sibling platforms already document this function (safe to bulk-approve)
- `confidence: medium` вАФ exactly 1 sibling platform confirms (review recommended)
- `confidence: low` вАФ no sibling confirmation; human review required before applying

```bash
./harness-bin annotate --config harness-config.yml          # generate patches
./harness-bin annotate --config harness-config.yml --apply  # apply to SDK source
```

The `id:` is inferred deterministically (`AmityPostRepository.createPost` вЖТ `post.create`).
A human or AI agent should review the patch file before applying (`id:` values may need adjustment),
then verify with `audit` to confirm the count drops.

### `serve` вАФ Live dashboard вЪЩпЄП Computational
Starts an HTTP server with a real-time findings dashboard.

```bash
./harness-bin serve --config harness-config.yml   # default port :8080
```

Endpoints:
- `GET /` вАФ dashboard HTML (open/fixed/needs_human counts + platform coverage bars)
- `GET /api/report` вАФ full `harness-report.json` as JSON
- `GET /api/coverage` вАФ per-platform annotated/total/percent: `{"platforms":[{"platform":"android","annotated":599,"total":599,"percent":100},...]}`

## Harness Internal Architecture

```
internal/
вФЬвФАвФА scanner/     вЖР Parses begin_sample_code markers from SDK files вЖТ Snippet structs

вФЬвФАвФА migrator/    вЖР Adds imports + replaces CodeGroups in doc pages

вФФвФАвФА report/      вЖР Finding types, read/write harness-report.json
```

## Generated Snippet MDX Format

Each `snippets/*.mdx` file is AUTO-GENERATED:

~~~~mdx
{/* AUTO-GENERATED by harness gendocs вАФ do not edit manually */}
{/* Source: AmityClientLogin.kt, AmityClientLogin.swift, ... */}

\<CodeGroup>

```kotlin Android
fun login(userId: String) { ... }
```

```swift iOS
func login(userId: String) { ... }
```

```dart Flutter
void login(String userId) { ... }
```

```typescript TypeScript
async function login(userId: string) { ... }
```

\</CodeGroup>
~~~~

Doc pages import and use them:

~~~~mdx
import LoginUser from '/snippets/social-plus-sdk/getting-started/login-user.mdx';

### Step 2: Login User

\<LoginUser />
~~~~

## Autonomous AI Agent Operation

The harness is designed so an AI agent can run the entire workflow autonomously,
with no human intervention between phases. `harness-tasks.md` is the AI's runbook.

```
Computational steps вФАвФАвЖТ produce findings / artifacts
                                  вЖУ
Inferential steps вФАвФАвЖТ AI agent reads harness-tasks.md, fills gaps
                                  вЖУ
Computational steps вФАвФАвЖТ verify, regenerate, advance to next phase
```

### Phase 0 вАФ SDK Annotation Campaign (in progress)
*Add `begin_public_function` markers to all public SDK functions that are missing them.*

| Step | Type | Who |
|---|---|---|
| `audit` вЖТ find `PUBLIC_FUNC_UNANNOTATED` | вЪЩпЄП Computational | harness |
| `annotate` вЖТ generate `annotation-patches.yml` | вЪЩпЄП Computational | harness |
| Review `id:` values in patch file for correctness | рЯ§Ц Inferential | AI agent / human |
| `annotate --apply` вЖТ insert markers into SDK source | вЪЩпЄП Computational | harness |
| `audit` вЖТ verify count drops | вЪЩпЄП Computational | harness |

**Exit criteria:** 0 `PUBLIC_FUNC_UNANNOTATED` findings

### Phase 1 вАФ Migration (one-time, in progress)
*Convert all existing hardcoded `<CodeGroup>` blocks to generated imports.*

| Step | Type | Who |
|---|---|---|
| `audit` вЖТ find `DOC_PAGE_STALE_IMPORT` | вЪЩпЄП Computational | harness |
| `gendocs --clean` вЖТ generate snippet MDX files | вЪЩпЄП Computational | harness |
| `migrate` вЖТ replace CodeGroups with imports | вЪЩпЄП Computational | harness |
| Create page manifests (read each MDX, infer sections + functions) | рЯ§Ц Inferential | AI agent |
| Section-targeted `migrate` using manifests | вЪЩпЄП Computational | harness |
| Verify Mintlify build passes | вЪЩпЄП Computational | CI |

**Exit criteria:** 0 `DOC_PAGE_STALE_IMPORT` findings with status `open`

### Phase 2 вАФ Coverage (ongoing)
*Ensure every manifest function has a snippet in every platform SDK.*

| Step | Type | Who |
|---|---|---|
| `audit` вЖТ find `MISSING_SNIPPET` per manifest | вЪЩпЄП Computational | harness |
| `prompt` вЖТ generate `harness-tasks.md` | вЪЩпЄП Computational | harness |
| Find function in SDK source or determine it doesn't exist | рЯ§Ц Inferential | AI agent |
| Write `begin_sample_code` snippet wrapping real SDK function | рЯ§Ц Inferential | AI agent |
| Mark as `DOES_NOT_EXIST` if function not available on platform | рЯ§Ц Inferential | AI agent |
| `audit` вЖТ verify 0 new MISSING_SNIPPET findings | вЪЩпЄП Computational | harness |
| Compile snippets (verify they build) | вЪЩпЄП Computational | harness |
| `gendocs` вЖТ regenerate snippet MDX | вЪЩпЄП Computational | harness |

**Exit criteria:** Every manifest function has snippets or is marked `DOES_NOT_EXIST`

### Phase 3 вАФ Steady State (CI, future)
*Keep docs in sync automatically on every SDK change.*

| Step | Type | Who |
|---|---|---|
| SDK PR merged вЖТ trigger harness | вЪЩпЄП Computational | CI |
| `audit` вЖТ detect changes | вЪЩпЄП Computational | harness |
| `gendocs` вЖТ regenerate changed snippets | вЪЩпЄП Computational | harness |
| Open docs PR if diff exists | вЪЩпЄП Computational | CI |
| Mintlify preview build validates | вЪЩпЄП Computational | CI |

**No inferential steps in steady state** вАФ the system is fully automated once manifests and
snippets are complete.

## Full Pipeline Commands

```bash
# вЪЩпЄП COMPUTATIONAL вАФ run these with the harness binary

# 1. Find all problems
./harness-bin audit --config harness-config.yml

# 2. Auto-fix broken sp_docs_page URLs (computational; remaining go to needs_human)
./harness-bin fix --config harness-config.yml

# 3. Regenerate all snippet MDX files
./harness-bin gendocs --config harness-config.yml --clean

# 4. Re-audit to get fresh DOC_PAGE_STALE_IMPORT list
./harness-bin audit --config harness-config.yml

# 5. Migrate doc pages to use generated snippets
./harness-bin migrate --config harness-config.yml

# 6. Generate AI agent runbook (computational output вЖТ triggers inferential work)
./harness-bin prompt --config harness-config.yml
# вЖТ produces harness-tasks.md

# рЯ§Ц INFERENTIAL вАФ AI agent reads harness-tasks.md and:
#   - Creates page manifests (read each MDX, infer sections + function keys)
#   - Writes missing snippets (look up SDK source, wrap in begin_sample_code)
#   - Marks DOES_NOT_EXIST for functions unavailable on a platform
#   - Resolves needs_human ASC_PAGE_INVALID findings

# вЪЩпЄП COMPUTATIONAL вАФ back to harness after inferential work:
./harness-bin audit --config harness-config.yml  # verify 0 open findings
./harness-bin gendocs --config harness-config.yml --clean  # regenerate with new snippets
```
