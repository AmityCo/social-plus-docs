## SSO SAML Configuration

For organizations requiring enterprise-grade security and user management:

### Prerequisites

<AccordionGroup>
  <Accordion title="Technical Requirements" icon="gear">
    - **Identity Provider**: Active SAML 2.0 compatible IDP
    - **Domain Control**: Verified organizational domain
    - **User Attributes**: Email address mapping capability
    - **Security Policies**: SAML assertion and encryption support
  </Accordion>
  
  <Accordion title="Access Planning" icon="users">
    - **User Scope**: Determine organization-wide vs. selective access
    - **Role Mapping**: Plan user role assignments
    - **Transition Strategy**: Migrate existing users to SSO
    - **Backup Access**: Maintain emergency access procedures
  </Accordion>
</AccordionGroup>

### Configuration Steps

<Steps>
  <Step title="Request SSO Setup">
    Contact our [support team](mailto:support@social.plus) with your SSO requirements:
    
    - Organization details and domain
    - User access scope (all users vs. selective)
    - Preferred implementation timeline
    - Technical contact information
  </Step>
  
  <Step title="Configure Your IDP">
    Set up SAML integration with these parameters:
    
    <CodeGroup>
      ```xml SAML Configuration
      <!-- SAML Response URL -->
      <saml:Destination>
        https://amity-portal-prod-auth.amity.co/saml2/idpresponse
      </saml:Destination>
      
      <!-- Audience Restriction -->
      <saml:AudienceRestriction>
        <saml:Audience>
          urn:amazon:cognito:sp:eu-central-1_gcJoCA15K
        </saml:Audience>
      </saml:AudienceRestriction>
      
      <!-- Email Attribute Mapping -->
      <saml:Attribute Name="http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress">
        <saml:AttributeValue>user@organization.com</saml:AttributeValue>
      </saml:Attribute>
      ```
      
      ```json Identity Provider Settings
      {
        "samlResponseUrl": "https://amity-portal-prod-auth.amity.co/saml2/idpresponse",
        "audienceRestriction": "urn:amazon:cognito:sp:eu-central-1_gcJoCA15K",
        "attributeMapping": {
          "email": "http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress"
        }
      }
      ```
    </CodeGroup>
  </Step>
  
  <Step title="Test Integration">
    Validate the SSO setup with test users before full deployment
  </Step>
  
  <Step title="Production Rollout">
    Enable SSO for your organization and communicate changes to users
  </Step>
</Steps>

### SSO Access Flow

<img src="../../.gitbook/assets/Screenshot 2567-11-13 at 11.31.59 AM.png" alt="SSO Login Flow" />

Once SSO is enabled:

1. **Domain Authentication**: Users enter organization domain
2. **IDP Redirect**: Automatic redirect to your identity provider
3. **SAML Assertion**: IDP validates user and sends SAML response
4. **Portal Access**: User gains access to social.plus Portal and Console

## Post-SSO Integration Changes

<Warning>
**Important Changes**: Enabling SSO SAML affects user management and access controls across your organization.
</Warning>

<AccordionGroup>
  <Accordion title="User Migration" icon="user-arrow-right">
    **Automatic Migration**
    - Existing portal users automatically migrated to SSO
    - User roles and permissions preserved
    - Seamless transition with no data loss
    
    **New User Provisioning**
    - Automatic account creation for SSO users
    - Portal and Console access granted simultaneously
    - Role assignment based on organizational mapping
  </Accordion>
  
  <Accordion title="Access Control Changes" icon="shield">
    **Console Admin Creation**
    - Direct admin user creation disabled
    - All user access managed through SSO
    - Enhanced security through centralized control
    
    **Legacy User Management**
    - Pre-existing admin users remain functional
    - Recommended to migrate to SSO for consistency
    - Optional removal for full SSO enforcement
  </Accordion>
</AccordionGroup>
